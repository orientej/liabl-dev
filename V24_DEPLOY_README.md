## Pre-Deploy Checklist

Do these in order **before** pushing code to `liabl-ai`.

### 1. Verify Supabase state

Confirm migrations 001 (initial schema) and 002 (M1 fixes) are already applied. If your Supabase project has real signed waivers or participants from testing, they should have `dob` as a proper `date` type and `answers.healthStatus` as an array.

### 2. Apply migrations 003 ‚Üí 006 in order

Open the Supabase SQL editor and run these four migrations, in this exact order. Each one is idempotent-friendly but assumes the prior ones are applied.

| Order | File | What it does |
|---|---|---|
| 1 | `supabase/migrations/003_m2_incidents.sql` | Adds `notified_by` column to `incidents`, adds name-required constraint |
| 2 | `supabase/migrations/004_m2_hashing.sql` | Adds `document_hash` and `pdf_url` columns to `waivers` with an index |
| 3 | `supabase/migrations/005_m2_audit_events.sql` | Creates `audit_events` table with indexes and RLS |
| 4 | `supabase/migrations/006_m2_notifications.sql` | Creates `notifications` table + three Postgres triggers that auto-generate notifications from waiver/incident writes |

Copy each file's contents into the SQL editor and run. If any migration fails, stop and inspect ‚Äî do not proceed to the next one.

### 3. Create the Supabase Storage bucket

This is required for T3 (sealing). Without it, waiver signings will save successfully but sealing will fail silently and log the error.

In the Supabase dashboard:

1. Go to Storage ‚Üí Create new bucket
2. Name: **`waivers`** (exactly this)
3. Public: **No** (private bucket, signed URLs only)
4. Click Create

Then set the bucket policy ‚Äî go to Storage ‚Üí Policies ‚Üí New policy for the `waivers` bucket:

```sql
create policy "allow_upload_waivers" on storage.objects
  for insert with check (bucket_id = 'waivers');
```

**Do not** add UPDATE or DELETE policies. Their absence enforces PDF immutability at the storage layer ‚Äî once a waiver is sealed, its PDF cannot be modified.

### 4. Confirm environment variables

The environment variables in `liabl-ai` Vercel project should already be set from v23. Verify:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

Nothing new is required for v24.

## Post-Deploy Smoke Test

Run this test after deploy to confirm the end-to-end flow works. This is where you find out if sealing actually works, if audit events land in the right table, and if notifications get generated by the Postgres triggers.

### 1. Complete a real waiver signing

1. Go to `liabl-ai.vercel.app/participant`
2. Complete the flow with real data (use your own email ‚Äî you'll want to see the confirmation state)
3. At the signature step, submit

### 2. Verify what should have happened

Open Supabase and check each of these:

**Table: `waivers`** ‚Äî new row should have:
- `signed_at` populated
- `ip_address` populated (a real IP, not null)
- `document_hash` populated (SHA-256 hex string)
- `pdf_url` populated (signed URL to Supabase Storage)

**Storage: `waivers` bucket** ‚Äî a PDF should exist at `waivers/YYYY/MM/<waiverId>.pdf`. Try opening the signed URL ‚Äî the PDF should render with the participant name, all clauses, the signature, and the SHA-256 hash in the footer.

**Table: `audit_events`** ‚Äî should contain five events for this waiver: `session.started`, `waiver.generated`, `document.viewed`, `waiver.signed`, `document.sealed`.

**Table: `notifications`** ‚Äî should contain a `waiver_signed` notification generated by the Postgres trigger (verify by refreshing the operator dashboard's Notifications tab).

### 3. Verify the operator dashboard

Go to `liabl-ai.vercel.app/operator` and check:

- **Roster tab:** the participant appears in the list with a real risk score (based on their disclosed conditions, age, activity). Click to expand ‚Äî the `WaiverDetail` panel should show real clauses, real IP, real hash, and a functional "Download PDF" button.
- **Notifications tab:** the `waiver_signed` notification appears. Click it ‚Äî it marks read.
- **Analytics tab:** the KPI cards and charts reflect the new signing (may need a refresh).
- **Incidents tab:** file a test incident linked to the waiver you just signed. Verify a row appears in the `incidents` table. Mark carrier notified ‚Äî verify `carrier_notified_at` and `notified_by` are populated.

If all of the above passes: v24 is deployed correctly.

## Known Limitations Carried Forward

These are called out in the M2 READMEs ‚Äî worth knowing they're not gaps you need to fix now:

- **Incident reference generator** ‚Äî `nextIncidentRef` in `lib/incidents.ts` uses a row count, which is not collision-safe under concurrent filings. Fine for pilot phase; replace with a database sequence before multiple operators use the system simultaneously.
- **Carrier notification is manual** ‚Äî the "Mark carrier notified" action logs a human action; no real webhook to K&K/Markel/etc. yet. Deferred to a later integration-dependent task.
- **`allow_all` RLS policies** ‚Äî still in place from v22. This is the Milestone 5 security pass. Not addressed in v24.
- **Avg sign time in Analytics** ‚Äî shown as `‚Äî`. Requires adding a `flow_started_at` column to `waivers`, small M3 addition.
- **Real-time push notifications** ‚Äî the Notifications tab loads on mount and has a manual Refresh. Supabase Realtime subscription is a future enhancement.
- **Email delivery** ‚Äî the participant confirmation page still says "emailed to you" but no email actually sends. Requires an email service integration.
- **Age distribution chart** ‚Äî only populated for waivers signed after M1 (post-DOB fix). Prior waivers won't have derivable ages from `answers.dob`.
- **Re-sealing old waivers** ‚Äî pre-v24 waivers have `document_hash = null` and `pdf_url = null`. A backfill script could re-seal them from stored fields; not included in v24.

## Troubleshooting

**Build fails on TypeScript error involving `WaiverDetailRow` or `session_id`:**
The T4 integration edits to `RosterTab.tsx` may not have applied cleanly. Compare against `Liabl-M2-T4/README.md` for the three required edits.

**First signing succeeds but no PDF appears in Storage:**
The `waivers` bucket doesn't exist or the storage policy is missing. See Pre-Deploy step 3.

**First signing succeeds but `document_hash` is null:**
Migration 004 didn't apply, or the sealing errored (check the browser console for the client-side error, or Supabase logs for the storage upload error). The waiver row is still saved ‚Äî this is the deliberate graceful-degradation design.

**No notifications generated after a signing:**
Migration 006 didn't apply, or the triggers didn't install. Check `information_schema.triggers` in the SQL editor for `waivers_notification_trigger` and `incidents_notification_trigger`.

**Type error about `ACTIVITY_LABELS`:**
Confirm `export const ACTIVITY_LABELS` exists at the top of `lib/document-engine.ts`. It should ‚Äî v23 has it ‚Äî but the T3 README flagged this as a thing to verify.
